# Architecture Decision Records (ADR)

> Архитектурные решения проекта SiMOTO-sklad, зафиксированные в формате ADR (Architecture Decision Record).

## Принципы

- **Lightweight формат**: Title, Status, Context, Decision, Consequences (5 секций)
- **Обязательные альтернативы**: каждое решение документирует рассмотренные варианты
- **Статусы**: proposed → accepted → deprecated/superseded
- **Связи**: ADR ссылаются на задачи (PLAN.md) и другие ADR

## Lifecycle ADR

### proposed (Предложено)
- Решение рассматривается
- Альтернативы документированы
- Ожидает утверждения перед реализацией

### accepted (Принято)
- Решение утверждено
- Реализация может начаться
- Является текущим стандартом

### deprecated (Устарело)
- Решение больше не рекомендуется
- Существующие реализации могут остаться
- Новый код не должен следовать этому паттерну

### superseded (Заменено)
- Решение заменено более новым ADR
- Ссылка на заменяющий ADR обязательна
- Существующие реализации должны мигрировать

## Как читать ADR

1. **Context** — проблема или потребность, вызвавшая решение
2. **Decision** — что было решено и почему
3. **Alternatives Considered** — что рассматривалось и почему отклонено
4. **Consequences** — положительные и отрицательные последствия

```markdown
## Context

## Decision
{Решение и обоснование}

## Alternatives Considered

- **Cons**: ...

- ...

### Negative
- ...

## Implementation Notes
{Технические детали реализации}
```
