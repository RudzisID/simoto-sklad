# Сравнение скана поставок с отчётом Ozon

> Создано: 2026-06-17
> Назначение: документация алгоритма сверки скана Склад.xlsx с отчётом Ozon,
> описание найденных расхождений и план реализации функции в приложении.

## 1. Структура входных данных

### 1.1 Скан.xlsx (из МойСклад)

Лист: `Лист1`

Структура: строки данных чередуются с пустыми/служебными строками (каждая вторая строка — данные).

| Визуальная колонка | Индекс (0‑based) | Название | Тип | Пример |
|---|---|---|---|---|
| A | 0 | (служебная) | — | — |
| B | 1 | Номер заказа покупателя | string (может быть двухстрочным) | `0135812416-0086-2\nOzon: ii11438790761` |
| C | 2 | (внутренний ID МойСклад) | number | `77864` |
| D | 3 | **Сумма заказа** | string с ₽ | `"5220 ₽"` |
| E | 4 | (ID документа?) | number | `77575` |
| F | 5 | Возврат (частичный) | string с ₽ или `"—"` | `"3570 ₽"` |
| G | 6 | **Реальный платёж** | string (может быть двухстрочным) или `"—"` | `"1650 ₽\n1650 ₽ (Ozon)"` |
| H | 7 | Статус | многострочный string | `"Частичная отмена⟳\nДоставлен → Возврат (Ozon)\n↳ Возврат: ..."` |
| I | 8 | (резерв) | — | — |
| J | 9 | (резерв) | — | — |

**Номер заказа — первая строка колонки B.** Вторая строка (`Ozon: ii...`) — ID возврата Ozon, относится к тому же заказу, не является отдельным заказом.

Статусы (определяются по первой строке H):

| Статус | Кол-во | Категория | F | G |
|---|---|---|---|---|
| `Отгружен` + `Доставлен (Ozon)` | 512 | Нормальный | = D | `"—"` (0) |
| `Отменен` + `Доставлен → Возврат` | 10 | Полный возврат | `"—"` (0) | = D |
| `Возврат` + `Доставлен → Возврат` | 3 | Полный возврат | `"—"` (0) | = D |
| `Частичная отмена` + `Доставлен → Возврат` | 3 | Частичный возврат | >0 | = D − F |
| `Отгружен` (без Доставлен) | 1 | Тестовый | = D | `"—"` (0) |

### 1.2 Отчёт Ozon (Отчет по товарам за период ...)

Лист: `Начисления`

Всего строк: ~3150 (с заголовком).

Структура (колонки A–Y, но ключевые первые 10):

| Колонка | Индекс | Название | Назначение |
|---|---|---|---|
| A | 0 | Номер строки | Порядковый номер |
| B | 1 | Операция | Тип операции (текст на русском) |
| C | 2 | **Номер заказа** | Номер заказа для сопоставления |
| D | 3 | Дата | Дата операции (число Excel) |
| E | 4 | Склад | Тип склада (FBS, etc.) |
| F | 5 | ID отправления | Внутренний ID Ozon |
| G | 6 | Номер товара | Артикул/код товара |
| H | 7 | Наименование товара | Описание товара |
| I | 8 | Кол-во | Количество единиц |
| J | 9 | **Цена (колонка для сравнения)** | Сумма операции (продажи/возврата) |

**Важно:** колонка J содержит ТОЛЬКО суммы продаж (`Поставка покупателю`). Операции возврата (`Поставка и обработка возврата`) имеют J = 0.

**Операции (колонка B), определяющие J:**
- `Поставка покупателю` — J = цена проданного товара
- `Продажа бренда` — J = 0 (доп. услуга)
- `Подписка` — J = 0
- `Комиссия` — J = 0
- `Поставка и обработка возврата, отмены, невыкупа` — J = 0
- `Плата за эквайринг` — J = 0

### 1.3 Парсинг числа (функция `parseSum`)

```javascript
function parseSum(v) {
  if (v == null || v === '—' || v === '-') return null;
  if (typeof v === 'number') return v;
  const firstLine = String(v).split(/[\n\r]/)[0].trim(); // только первая строка!
  const s = firstLine.replace(/[^0-9.,-]/g, '').replace(',', '.');
  const n = parseFloat(s);
  return isNaN(n) ? null : n;
}
```

**Важно:** колонка G и B могут содержать перевод строки `\n`. Без `split('\n')[0]` числа склеиваются (напр. `"1650 ₽\n1650 ₽ (Ozon)"` → `16501650` вместо `1650`).

## 2. Как было найдено расхождение

### 2.1 Входные данные

- **Скан.xlsx**: 529 заказов, сумма колонки D = **1 644 657 ₽**
- **Отчёт Ozon**: все строки колонки J+ (только положительные) = **1 630 294 ₽**

### 2.2 Первое сравнение: Скан D vs Отчёт J+

```
Скан D (все 529)    = 1 644 657
Отчёт J+ (все)      = 1 630 294
Разница             =   14 363
```

### 2.3 Уточнение: что попало в отчёт

В отчёте Ozon есть **495 заказов, которых нет в скане** (чужие/другие поставки). Их J+ = 5 507 ₽.

Пользователь считает: J+ по своим заказам + тестовый заказ:
```
1 630 294 (J+ все)
−   5 507 (чужие заказы в отчёте)
+  10 500 (тестовый заказ 99999999-9999-19, нет в отчёте)
= 1 635 287 (сумма пользователя)
```

### 2.4 Разбивка по категориям

**Нормальные (512 заказов):**
```
Скан D  = 1 513 126
Отчёт J+ = 1 513 126
Разница = 0 ✅ (каждый заказ сходится индивидуально)
```

**Полные возвраты (13 заказов):**
```
Скан D  = 101 993
Скан G  = 101 993
Отчёт J+ = 101 993
Разница = 0 ✅ (J+ = G, J всего = 18 853 — остальное ушло в J-)
```

**Частичные возвраты (3 заказа) — ИСТОЧНИК РАСХОЖДЕНИЯ:**

| Заказ | D (скан) | F (возврат) | G (платёж) | J+ (отчёт) | Разрыв | Причина |
|---|---|---|---|---|---|---|
| 0135812416-0086-2 | 5 220 | 3 570 | 1 650 | 3 570 | **1 650** | J+ = F (возврат), G нет в отчёте |
| 0203476918-0059-1 | 7 028 | 5 508 | 1 520 | 4 548 | **2 480** | J+ ≠ F и ≠ G |
| 86404268-0161-2 | 6 790 | 1 550 | 5 240 | 1 550 | **5 240** | J+ = F (возврат), G нет в отчёте |
| **Итого** | **19 038** | **10 628** | **8 410** | **9 668** | **9 370** | |

**Тестовый заказ (1):**
```
99999999-9999-19: D = 10 500 — нет в отчёте Ozon
```

### 2.5 Корневая причина

Отчёт Ozon **не показывает возвращённые суммы в колонке J**. Для частичных возвратов J содержит только стоимость **проданных** (не возвращённых) позиций. Скан же показывает **полную сумму заказа** D, включая возвращённые товары.

Формула расхождения для частичного возврата:
```
Разрыв = D − J+ = (F + G) − J+ = G (если J+ = F, как в 2 из 3 случаев)
```

### 2.6 Итоговый баланс

```
Скан D всего           = 1 644 657
Отчёт J+ свои заказы   = 1 624 787
+ тестовый заказ       =   10 500
= сумма пользователя   = 1 635 287

Расхождение: 1 644 657 − 1 635 287 = 9 370
```

## 3. Алгоритм сверки (для реализации)

### 3.1 Вход

1. **Скан** — массив объектов, полученных после поиска по кнопке «Поиск» (данные из МойСклад)
2. **Отчёт** — XLSX-файл, загруженный пользователем (лист `Начисления`)

### 3.2 Парсинг отчёта

```javascript
function parseReport(file) {
  const wb = XLSX.read(file.data, { type: 'array' });
  const ws = wb.Sheets['Начисления'];
  const data = XLSX.utils.sheet_to_json(ws, { header: 1, defval: '' });

  const reportMap = {}; // orderNum → { jTotal, jPos, jNeg, rows }
  for (let i = 1; i < data.length; i++) {
    const orderNum = String(data[i][2] || '').trim(); // колонка C
    if (!orderNum) continue;
    const j = typeof data[i][9] === 'number' ? data[i][9] : 0; // колонка J
    if (!reportMap[orderNum]) reportMap[orderNum] = { jTotal: 0, jPos: 0, jNeg: 0 };
    reportMap[orderNum].jTotal += j;
    if (j > 0) reportMap[orderNum].jPos += j;
    else reportMap[orderNum].jNeg += j;
  }
  return reportMap;
}
```

### 3.3 Сопоставление по номеру заказа

Ключ сопоставления — **номер заказа** (первые символы до `\n` в колонке B скана / колонка C в отчёте).

```javascript
function getKey(rawB) {
  return String(rawB).split('\n')[0].trim();
}
```

### 3.4 Категоризация и сравнение

```javascript
function categorizeOrder(statusH, sumD, retF, payG) {
  const isPartial = statusH.includes('Частичная отмена');
  const isReturn = (statusH.includes('Возврат') || statusH.includes('Отменен')) && !isPartial;
  return { isPartial, isReturn };
}

function compareOrder(scanOrder, reportEntry) {
  const { sumD, retF, payG, isPartial, isReturn } = scanOrder;
  const jPos = reportEntry ? reportEntry.jPos : 0;
  const jTotal = reportEntry ? reportEntry.jTotal : 0;

  if (!reportEntry) {
    return { status: 'not-in-report', diff: sumD };
  }
  if (isPartial) {
    // Для частичных: J+ должен равняться G (факт. платежу), но Ozon показывает J+ = F (возврат)
    const diff = sumD - jPos; // ожидаемое расхождение
    return { status: 'partial', diff, note: 'J+ может не включать G' };
  }
  if (isReturn) {
    const diff = Math.abs(jTotal); // J total должен быть ≈ 0
    return { status: 'return', diff, note: 'J всего ≈ 0 для полных возвратов' };
  }
  // Нормальные
  const diff = sumD - jPos;
  return { status: diff === 0 ? 'ok' : 'mismatch', diff };
}
```

### 3.5 Вывод результатов

- **Зелёный** — совпало (diff = 0)
- **Жёлтый** — расхождение в пределах допустимого (частичные возвраты)
- **Красный** — не найдено в отчёте / значительное расхождение

### 4.2 Логика `runComparison()`

1. Прочитать выбранный XLSX через `XLSX.read()`
2. Построить `reportMap` (номер заказа → J+)
3. Пройти по `window.scannedOrders` (данные из поиска МойСклад)
4. Для каждого заказа:
   - Извлечь номер заказа (первая строка `shipmentNum`)
   - Найти в `reportMap`
   - Определить категорию по статусу
   - Вычислить разницу
   - Записать результат в `order.compareResult`
5. Вызвать `renderComparisonResults()`

## 5. Известные особенности

1. **Нормальные заказы (512 шт):** D = J+ в 100% случаев, расхождений нет
2. **Полные возвраты (13 шт):** J+ = G (сумма возврата), J всего ≈ 18 853 (не 0 из-за доп. услуг в отчёте)
3. **Частичные возвраты (3 шт):** всегда дают расхождение, т.к. J отчёта не включает G (факт. платёж)
4. **Тестовый заказ 99999999-9999-19:** нет в отчёте Ozon, D = 10 500
5. **Заказы с ii-номерами:** `Ozon: ii...` во второй строке B — это ID возврата, не отдельный заказ
6. **Разделители строк:** колонки B и G содержат `\n`, парсер должен брать только первую строку
